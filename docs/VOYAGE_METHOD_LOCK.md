## 1. Purpose

This document canonicalizes the Voyage Framework software-development methodology. It defines:

- the end-to-end process pipeline (intake → archive);
- the 16 methodology roles and their responsibilities;
- the artifact contracts produced at each step;
- the handoff protocol between roles;
- the relationship between methodology roles and the v4.3.0 runtime infrastructure;
- the multi-project model and workspace boundary rules.

This is a **methodology lock**, not an implementation contract. For runtime architecture see
`docs/VOYAGE_V4_1_CONTRACT.md`. For AI agent operating rules see `AGENTS.md`.

## 2. Core Boundaries

Voyage is not an autonomous AI agent framework and does not call models automatically.

The Voyage Framework system is divided into four boundary layers. These must never be conflated.

### 2.1 Voyage Framework Core (`voyage_framework/`)

The installed Python package. Contains:

- `TaskParser` / `TaskYamlSpec` — canonical task specification
- `TaskEngine` / `TaskRecord` — runtime task state (SQLite)
- `EventEngine` — append-only audit log
- `AgentRegistry` — read-only role profile catalog (6 runtime roles defined as of v4.3.0)
- `ModeRegistry` — read-only mode catalog (6 modes defined as of v4.3.0)
- `PromptGenerator` — deterministic prompt package builder
- `ContextBuilder` — context aggregation and comparison
- `Chronicler` — process documentation helpers
- `CLI` — `voyage` command surface

**Rules:**
- Does NOT store project-specific artifacts.
- Does NOT call AI models.
- Does NOT execute agents.
- Is versioned and released independently of any managed project.

### 2.2 Managed Project Workspace (`.voyage/`)

Per-project state directory, analogous to `.git/`. Contains:

- `project.yaml` — project identity and framework version
- `events.db` / `events.jsonl` — append-only audit log for this project
- `tasks.db` — SQLite task state for this project
- Generated artifacts: `CONTEXT.json`, `TASK.md`

**Rules:**
- Owned by the managed project, not the framework.
- Generated artifacts (`CONTEXT.json`, `TASK.md`) must NEVER be parsed as canonical truth.
- The `events.db` and `events.jsonl` files are live and must NEVER be edited manually.

### 2.3 External AI Executor

The human-mediated AI tool (Claude, Codex, Gemini, or similar) that receives a `PromptPackage`
and produces proposed results. The framework generates input for it; it does not call it.

**Rules:**
- Receives: `PromptPackage` (generated by `PromptGenerator`)
- Returns: proposed files, text, or decisions — not committed automatically
- Human review is required before any result enters version control

### 2.4 Dashboard (Future Read-Only Layer)

Not yet implemented in v4.3.0. When built, the dashboard must be:

- read-only over the `.voyage/` workspace;
- a visualization layer only — never a state mutator;
- decoupled from framework core via adapter contract (v4.2 milestone, deferred).

## 3. Multi-Project Model

Voyage operates as a global tool managing multiple per-project workspaces, analogous to the
`git` / `.git/` relationship.

| Analogy | Voyage | Git |
|---------|--------|-----|
| Global tool | `voyage` CLI | `git` CLI |
| Per-project state | `.voyage/` directory | `.git/` directory |
| Canonical specification | `task.yaml` | commit object |
| Append-only log | `events.jsonl` / `events.db` | reflog |
| Runtime state | `tasks.db` (SQLite) | index |

**Consequences:**

1. The framework core (`voyage_framework/`) must contain no project-specific data.
2. Each managed project initializes its own `.voyage/` workspace via `voyage init` (or equivalent bootstrap command).
3. Multiple projects can be active simultaneously with independent `.voyage/` state.
4. Framework upgrades must not invalidate existing `.voyage/` workspaces (migration path required per major version bump).

## 4. Process Map

The Voyage methodology pipeline has 12 sequential steps (0–11) followed by a repeating execution
cycle.

### 4.1 Methodology Pipeline (Steps 0–11)

| Step | Phase | Role | Primary Output |
|------|-------|------|----------------|
| 0 | Intake | — (human) | `00-project-intake.json` |
| 1 | Requirements Interview | Interviewer | `01-discovery.json` |
| 2 | Business Analysis | Business Analyst | `02-business.json` |
| 3 | UX Architecture | UX Architect | `03-ux.json` |
| 4 | Domain Architecture | Domain Architect | `04-domain.json` |
| 5 | Event Storming | Event Stormer | `05-events.json` |
| 6 | Feature Architecture | Feature Architect | `06-features.json` |
| 7 | Solution Architecture | Solution Architect | `07-solution.json` |
| 8 | Consistency Validation | Consistency Validator | `08-validation.json` |
| 9 | MVP Optimization | MVP Optimizer | `09-mvp.json` |
| 10 | Voyage Architecture | Voyage Architect | `CONTEXT.json`, `RULES.md`, `ADR/` |
| 11 | Task Generation | Task Generator | `TASKS/`, `task.yaml` |

### 4.2 Execution Cycle (Steps 12–16)

After step 11, each task enters the repeating execution cycle:

| Step | Phase | Role | Gate |
|------|-------|------|------|
| 12 | Implementation | Developer | Code and tests meet acceptance criteria |
| 13 | Review | Reviewer | Reviewer verdict A or B |
| 14 | Testing / QA | Tester | Quality gates pass; evidence reproducible |
| 15 | Closure Audit | Auditor | Phase closure audit complete; verdict issued |
| 16 | Chronicle | Chronicler | Process documentation committed |

### 4.3 Phase Lifecycle (per individual phase)

Each phase within the execution cycle (steps 12–16) follows the 8-step lifecycle defined in
`docs/guides/PHASE_WORKFLOW_POLICY.md`:

```
1. Contract     — define acceptance criteria
2. Prompt       — generate PromptPackage
3. Review       — approve prompt before execution
4. Implement    — external AI tool produces proposed result
5. Self-audit   — AI self-checks against acceptance criteria
6. Human review — human reviewer issues verdict (A/B/C/D)
7. Merge + tag  — authorized human merges and tags
8. Closure audit — Auditor signs off; EventEngine records closure
```

## 5. Role Catalog

The Voyage methodology defines **16 methodology roles**. These are process roles — they describe
what a human or AI agent must do at each step. They are distinct from the 6 runtime roles
currently registered in `AgentRegistry`.

> **Implementation status as of v4.3.0:** The `AgentRegistry` contains 6 runtime role profiles:
> `architect`, `developer`, `reviewer`, `qa`, `security`, `devops`. The 16 methodology roles
> listed below are not yet statically registered as `RoleProfile` objects. Future work
> (VOYAGE-METHOD-LOCK-2+) will map methodology roles to `RoleProfile` definitions.

### Phase 1 — Discovery

| # | Role ID | Display Name | Primary Responsibility |
|---|---------|--------------|------------------------|
| 1 | `interviewer` | Interviewer | Elicit raw project requirements via structured interview; produce discovery artifact |
| 2 | `business_analyst` | Business Analyst | Convert raw discovery into structured business model; identify stakeholders and constraints |

### Phase 2 — Design

| # | Role ID | Display Name | Primary Responsibility |
|---|---------|--------------|------------------------|
| 3 | `ux_architect` | UX Architect | Define user experience model, flows, and interaction contracts |
| 4 | `domain_architect` | Domain Architect | Model the domain: entities, aggregates, bounded contexts |
| 5 | `event_stormer` | Event Stormer | Identify domain events, commands, and projections |
| 6 | `feature_architect` | Feature Architect | Define feature catalog with acceptance criteria and dependencies |

### Phase 3 — Architecture

| # | Role ID | Display Name | Primary Responsibility |
|---|---------|--------------|------------------------|
| 7 | `solution_architect` | Solution Architect | Design end-to-end technical solution; choose stack and integration points |
| 8 | `consistency_validator` | Consistency Validator | Cross-check all design artifacts for contradictions and gaps |
| 9 | `mvp_optimizer` | MVP Optimizer | Reduce scope to minimum viable product; define MVP boundary |

### Phase 4 — Voyage Planning

| # | Role ID | Display Name | Primary Responsibility |
|---|---------|--------------|------------------------|
| 10 | `voyage_architect` | Voyage Architect | Produce `CONTEXT.json`, `RULES.md`, and `ADR/` for the managed project |
| 11 | `task_generator` | Task Generator | Generate canonical `task.yaml` files and `TASKS/` backlog |

### Phase 5 — Execution

| # | Role ID | Display Name | Primary Responsibility |
|---|---------|--------------|------------------------|
| 12 | `developer` | Developer | Implement requested behavior with a narrow, tested patch |
| 13 | `reviewer` | Reviewer | Review diffs for correctness, regressions, and criteria coverage |
| 14 | `tester` | Tester | Establish reproducible test evidence; exercise edge cases and quality gates |
| 15 | `auditor` | Auditor | Sign off phase closure; validate evidence; issue verdict |
| 16 | `chronicler` | Chronicler | Document the process; produce journals, decision logs, and tutorials |

## 6. Artifact Contracts

Each methodology step produces a canonical artifact. Steps 1–9 produce project-owned JSON files.
Framework core artifacts (`CONTEXT.json`, `TASK.md`) are generated and must not be parsed as
canonical truth.

| Artifact | Owner | Step | Status | Notes |
|----------|-------|------|--------|-------|
| `00-project-intake.json` | Project | 0 | Schema TBD | Human-authored project intake |
| `01-discovery.json` | Project | 1 | Schema TBD | Raw requirements from interview |
| `02-business.json` | Project | 2 | Schema TBD | Business model and stakeholder map |
| `03-ux.json` | Project | 3 | Schema TBD | UX flows and interaction contracts |
| `04-domain.json` | Project | 4 | Schema TBD | Domain model: entities, aggregates |
| `05-events.json` | Project | 5 | Schema TBD | Domain events, commands, projections |
| `06-features.json` | Project | 6 | Schema TBD | Feature catalog with acceptance criteria |
| `07-solution.json` | Project | 7 | Schema TBD | Technical solution and stack |
| `08-validation.json` | Project | 8 | Schema TBD | Cross-check results and gap list |
| `09-mvp.json` | Project | 9 | Schema TBD | MVP boundary and exclusion list |
| `CONTEXT.json` | Generated | 10 | Live | Generated by `ContextBuilder`; NOT canonical truth |
| `RULES.md` | Project | 10 | Live | Project-specific rules and constraints |
| `ADR/` | Project | 10 | Live | Architecture Decision Records |
| `task.yaml` | Project | 11 | **Canonical** | Parsed by `TaskParser` into `TaskYamlSpec`; canonical truth |
| `TASKS/` | Project | 11 | Live | Task backlog directory |
| `TASK.md` | Generated | 12 | Generated | Generated from `TaskRecord`; NOT canonical truth |
| `events.jsonl` | Framework | All | Append-only | EventEngine dual-write log; must never be edited manually |
| `events.db` | Framework | All | Append-only | EventEngine SQLite store; must never be edited manually |

**Canonical source-of-truth hierarchy:**

```
task.yaml → TaskYamlSpec → TaskRecord → EventEngine
     ↑ canonical spec            ↑ append-only audit log

Generated (re-derivable, never parsed as truth):
  CONTEXT.json, TASK.md
```

## 7. Handoff Protocol

Each role-to-role transition follows a standard handoff protocol. The human operator (or
orchestrating agent) is responsible for verifying readiness criteria before initiating the
next step.

### 7.1 Standard Handoff Structure

```
Source role        — the role completing work
Target role        — the role beginning work
Handoff artifact   — what is passed to the target role
Readiness criteria — conditions that must be true before handoff proceeds
Stop conditions    — blockers that prevent handoff; work must return to source role
```

### 7.2 Handoff Table

| # | Source | Target | Artifact | Readiness | Stop Condition |
|---|--------|--------|----------|-----------|----------------|
| H01 | Interviewer | Business Analyst | `01-discovery.json` | All required questions answered; discovery JSON valid | Stakeholder undefined or scope unbounded |
| H02 | Business Analyst | UX Architect | `02-business.json` | Business goals stated; constraints documented | Business model internally contradictory |
| H03 | UX Architect | Domain Architect | `03-ux.json` | User flows complete; all interactions defined | Critical user journey undefined |
| H04 | Domain Architect | Event Stormer | `04-domain.json` | Entities and aggregates named; bounded contexts drawn | Domain model contradicts business model |
| H05 | Event Stormer | Feature Architect | `05-events.json` | All commands and events named; projections listed | Event storm incomplete for primary use case |
| H06 | Feature Architect | Solution Architect | `06-features.json` | Feature catalog complete; acceptance criteria on each feature | Any feature has no acceptance criterion |
| H07 | Solution Architect | Consistency Validator | `07-solution.json` | Stack chosen; integration points defined; risks noted | Architecture cannot support core features |
| H08 | Consistency Validator | MVP Optimizer | `08-validation.json` | All cross-artifact contradictions resolved or documented | Unresolved contradiction blocks implementation |
| H09 | MVP Optimizer | Voyage Architect | `09-mvp.json` | MVP boundary set; excluded features listed | MVP scope undefined or unbuildable |
| H10 | Voyage Architect | Task Generator | `CONTEXT.json`, `RULES.md`, `ADR/` | Context artifact valid; rules explicit | `CONTEXT.json` missing required fields |
| H11 | Task Generator | Developer | `task.yaml`, `TASKS/` | Each `task.yaml` passes `TaskParser`; `TaskRecord` created | `task.yaml` fails `TaskParser` validation |
| H12 | Developer | Reviewer | Implementation + tests | Acceptance criteria self-checked; quality gates run | Any acceptance criterion unmet |
| H13 | Reviewer | Tester | Reviewed implementation | Reviewer verdict A or B issued | Reviewer verdict C (conditional) or D (reject) |
| H14 | Tester | Auditor | Test results + evidence | All tests pass; edge cases exercised; results reproducible | Test failure unresolved or evidence missing |

### 7.3 Closure Handoff

After H14, the Auditor issues a closure verdict and the Chronicler documents the phase.
The EventEngine records the closure event. The task is then archived.

```
Auditor    → issues closure verdict (A/B/C/D) → EventEngine records PHASE_CLOSED
Chronicler → documents journal, decision log  → process artifact committed
TaskEngine → transitions TaskRecord to archived
```

## 8. PromptPackage Contract

Every execution step (12–16) that involves an external AI tool must produce a `PromptPackage`
via `PromptGenerator`. The package is the only authorized handoff vehicle between Voyage
Framework and an external AI executor.

### 8.1 Minimum PromptPackage Contents

```
1. Role Profile     — role_id, display_name, purpose, responsibilities, boundaries
2. Mode Profile     — mode id, instructions, constraints, output expectations, checklist
3. Task Context     — task id, title, description, acceptance criteria
4. Relevant Files   — file paths and content needed to complete the task
5. Instructions     — step-by-step task instructions
6. Output Contract  — expected outputs, format, forbidden actions
7. Stop Conditions  — conditions under which the AI must stop and report back
```

### 8.2 PromptPackage Rules

- Generated by `PromptGenerator` only — never constructed ad hoc.
- Must not contain secrets, credentials, or `.env` contents.
- Must include explicit stop conditions.
- Human review is required before the package is sent to an external tool.
- The external tool's output is a **proposed result** — it must not be committed without
  human review and authorized merge per the 8-step phase lifecycle.

## 9. Relationship to Existing v4.3.0 Infrastructure

This section maps methodology concepts to v4.3.0 runtime components.

### 9.1 Runtime Role Registry vs. Methodology Roles

The `AgentRegistry` (`voyage_framework/core/agent_registry.py`) currently defines 6 runtime
role profiles. The mapping to methodology roles is approximate — the runtime profiles cover
execution-phase concerns, not discovery or design phases.

| Runtime Role | Closest Methodology Role(s) |
|---|---|
| `architect` | Solution Architect, Consistency Validator, Voyage Architect |
| `developer` | Developer |
| `reviewer` | Reviewer |
| `qa` | Tester |
| `security` | Cross-cutting — applies at all execution phases |
| `devops` | Cross-cutting — applies to CI/CD and deployment phases |

The 16 methodology roles are NOT yet registered as `RoleProfile` objects in `AgentRegistry`.
Future work will extend the registry with full methodology role profiles or introduce a
`MethodologyRegistry` companion.

### 9.2 Mode Registry vs. Methodology Modes

The `ModeRegistry` (`voyage_framework/core/prompt_modes.py`) defines 6 modes:

| Mode | Primary Use in Methodology |
|------|---------------------------|
| `analysis` | Steps 1–9 (discovery, design, architecture) |
| `implementation` | Step 12 (development) |
| `review` | Step 13 (code review) |
| `qa` | Step 14 (testing) |
| `security` | Cross-cutting |
| `handoff` | All handoffs (H01–H14) |

### 9.3 TaskParser / TaskEngine

`task.yaml` (produced at step 11) is consumed by:

- `TaskParser` — validates and parses into `TaskYamlSpec` (immutable, frozen Pydantic model)
- `TaskEngine` — creates and mutates `TaskRecord` in SQLite

These are the sole authorized mutators of task state. No methodology role may update task state
except through `TaskEngine`.

### 9.4 EventEngine

The `EventEngine` is the append-only audit log for all methodology events. Every handoff,
verdict, and closure event must be recorded. The dual-write (`events.db` + `events.jsonl`)
ensures both queryability and portability.

### 9.5 Chronicler

The `Chronicler` subsystem produces process documentation from runtime state:

- `journal` — chronological development journal
- `replay` — event replay for audit or retrospective
- `decision_log` — architecture decisions and rationale
- `docs_builder` — structured documentation output
- `tutorial_generator` — learning artifacts from completed phases

## 10. Dashboard Implications

The dashboard is a **future read-only layer**. It is NOT part of v4.3.0.

When implemented, the dashboard must conform to these constraints:

1. **Read-only:** reads from `.voyage/events.db`, `.voyage/tasks.db`, and generated artifacts only.
2. **No state mutation:** the dashboard must never write to `task.yaml`, `tasks.db`, or `events.db`.
3. **Adapter contract:** the dashboard must communicate with framework core through a versioned
   adapter interface (deferred to v4.2 milestone / future phase).
4. **No methodology role:** the dashboard is a visualization tool, not a methodology participant.
5. **Decoupled release:** dashboard development must not require changes to `voyage_framework/` core.

Any dashboard work that requires modifying `voyage_framework/core/` violates this constraint
and requires a methodology amendment (VOYAGE-METHOD-LOCK-2).

## 11. Legacy and Reference Sources

### 11.1 SkillTracer v3.0 (Methodology Provenance)

The 16 methodology roles and the artifact chain (00–09) were validated through the
SkillTracer v3.0 project pipeline. The SkillTracer pipeline served as the proof-of-concept
for the 4-agent / 12-sub-role process now canonicalized as the Voyage Framework methodology.

**What is preserved from SkillTracer v3.0:**
- The 4-phase pipeline structure (Discovery → Design → Architecture → Voyage)
- The 01–09 artifact chain and naming convention
- The sub-role decomposition that maps to the 16 methodology roles

**What is NOT carried into framework core:**
- SkillTracer project-specific data (interviews, domain models, features)
- SkillTracer-specific agent names or prompt content
- Any production or server-side SkillTracer infrastructure

The framework methodology is derived from SkillTracer v3.0 learnings; it does not embed
SkillTracer data or depend on SkillTracer infrastructure.

### 11.2 Legacy Framework Modules

The following modules exist in the repository for historical reference only:

| Module | Status | Notes |
|--------|--------|-------|
| `voyage_framework/agents/` | Legacy | Historical multi-agent runtime; not v4.1 core contract |
| `voyage_framework/langgraph_tools/` | Legacy | Historical LangGraph integration; not v4.1 core contract |
| `voyage_framework_v4_mvp/` (nested) | Snapshot | Earlier parallel snapshot; do not modify without explicit instruction |

These modules must not be treated as current architecture or as source of truth.
They are preserved for reference only.
