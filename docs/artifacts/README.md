# Voyage Methodology Artifacts

> Document scope: methodology source artifacts `00-project-intake.json` through `09-mvp.json`.
> Executable schema source: `voyage_framework/methodology/artifact_models.py`.
> Framework version: v4.3.0+.

---

## Purpose

The 00–09 artifacts are **methodology source documents**. They carry the reasoning that moves a raw project idea through structured discovery, design, and architecture phases until an MVP scope is defined and ready for task generation.

These artifacts are authored by humans and AI roles during the methodology pipeline. They are **not** runtime execution context, prompt packages, or autonomous-agent instructions. They are not generated by the framework — they are the input that the framework's later stages (task generation, execution) depend on.

Every artifact in the chain preserves assumptions, open questions, and unresolved risks from earlier phases. Downstream roles consult earlier artifacts; they do not overwrite upstream decisions silently.

## Artifact Chain

The 10 artifacts form a sequential pipeline. Each phase produces one artifact that becomes input for the next phase.

```
00 Project Intake
    ↓ (human authored)
01 Discovery
    ↓ (Interviewer)
02 Business
    ↓ (Business Analyst)
03 UX
    ↓ (UX Architect)
04 Domain
    ↓ (Domain Architect)
05 Events
    ↓ (Event Stormer)
06 Features
    ↓ (Feature Architect)
07 Solution
    ↓ (Solution Architect)
08 Validation
    ↓ (Consistency Validator)
09 MVP
    ↓ (MVP Optimizer)
    → Voyage Architect (Step 10): produces CONTEXT.json, RULES.md, ADR/
    → Task Generator (Step 11): produces task.yaml, TASKS/
```

Each role reads all earlier artifacts before producing its own. The chain must be traversed in order — no phase may be skipped.

## CONTEXT.json Naming Collision

There are two distinct artifacts both historically called `CONTEXT.json`:

**1. Methodology CONTEXT.json (Step 10 — Voyage Architect)**
- Synthesized from the 00–09 artifact chain
- Represents the project knowledge context for task generation
- Produced by the Voyage Architect role after MVP scope is defined
- **Status: not yet implemented in v4.3.0** — defined in VOYAGE_METHOD_LOCK.md §4.1 as Step 10 output

**2. Runtime CONTEXT.json (ContextBuilder)**
- Generated by `ContextBuilder` from `task.yaml` + SQLite task state + event summary
- Used as input context for `PromptGenerator`
- Produced by `voyage sync build --file task.yaml --output CONTEXT.json`
- **Status: live in v4.3.0**
- Must not be parsed as canonical truth — it is a generated view, not a source document

Both artifacts are generated views. They belong to completely different pipeline stages and carry different semantics. The naming collision creates potential confusion in documentation and tooling.

**Resolution:** The naming decision is deferred to a future task: **VOYAGE-CONTEXT-NAMING-1**. No renaming is performed in this phase.

| Field | Type | Default | Notes |

### 00-project-intake.json

- **Model:** `ProjectIntakeArtifact`
- **Owner role:** `interviewer`
- **Phase:** `0`
- **Produces input for:** `01-discovery.json`
- **Purpose:** The human operator's raw project idea expressed as a structured intake. This is the only artifact authored directly by the human, not by an AI role. It captures the initial idea, who the users are, what problem is being solved, and what outcomes are desired.

**Required fields:**

| `problem_statement` | `str` | What problem this product solves |
| `desired_outcomes` | `tuple[str, ...]` | What outcomes the human wants to achieve (at least one) |

| Field | Type | Default | Description |
|---|---|---|---|

| `non_goals` | `tuple[str, ...]` | `()` | What this project explicitly should NOT do |
| `schema_version` | `str` | `"1.0"` | Inherited from base |

| `open_questions` | `tuple[str, ...]` | `()` | Inherited from base |
| `assumptions` | `tuple[str, ...]` | `()` | Inherited from base |

**AI filling guidance:** The human fills this artifact. An AI Interviewer role reads it at the start of Step 1. It must not modify or rewrite the human's raw idea — it uses it as the primary source to generate `01-discovery.json`.

| Field | Type | Default | Description |

**AI filling guidance:** Read `00-project-intake.json` in full. Sharpen the raw idea into a precise problem statement. Populate `success_criteria` with measurable, observable conditions. Record uncertainty in `open_questions`; record untested assumptions in `assumptions`.

| Field | Type | Default | Description |
|---|---|---|---|
| `domain_rules` | `tuple[str, ...]` | `()` | Business rules governing entity behavior |
| `schema_version` | `str` | `"1.0"` | Inherited from base |
| `status` | `str` | `"draft"` | Inherited from base |
| `open_questions` | `tuple[str, ...]` | `()` | Inherited from base |

### 08-validation.json

- **Model:** `ValidationArtifact`
- **Owner role:** `consistency_validator`
- **Phase:** `8`
- **Consumes:** All earlier artifacts `00-project-intake.json` through `07-solution.json`
- **Produces input for:** `09-mvp.json`

**Required fields:**

| Field | Type | Description |
|---|---|---|
| `title` | `str` | Artifact title |
| `summary` | `str` | Summary of validation findings |
| `consistency_findings` | `tuple[str, ...]` | All cross-artifact findings, including gaps and contradictions (at least one) |
| `verdict` | `str` | Overall verdict: `"approved"`, `"approved_with_conditions"`, or `"rejected"` |

**Optional fields (with defaults):**

| Field | Type | Default | Description |
|---|---|---|---|
| `risks` | `tuple[str, ...]` | `()` | Risks identified during cross-artifact review |
| `blockers` | `tuple[str, ...]` | `()` | Issues that must be resolved before proceeding to MVP |
| `schema_version` | `str` | `"1.0"` | Inherited from base |
| `status` | `str` | `"draft"` | Inherited from base |
| `open_questions` | `tuple[str, ...]` | `()` | Inherited from base |
| `assumptions` | `tuple[str, ...]` | `()` | Inherited from base |

**AI filling guidance:** Read all 7 earlier artifacts. Check for: entities in `04-domain.json` that have no corresponding feature in `06-features.json`; features in `must_have` that the architecture in `07-solution.json` cannot demonstrably support; user flows in `03-ux.json` with no domain events. If `blockers` is non-empty, `verdict` must be `"rejected"` or `"approved_with_conditions"`. Never issue `"approved"` if unresolved blockers exist.

### 09-mvp.json

- **Model:** `MVPArtifact`
- **Owner role:** `mvp_optimizer`
- **Phase:** `9`
- **Consumes:** All earlier artifacts, with primary focus on `06-features.json` and `08-validation.json`
- **Produces input for:** Voyage Architect (Step 10) — produces `CONTEXT.json`, `RULES.md`, `ADR/`
- **Purpose:** Defines the minimum viable product boundary. Selects the smallest slice of features and acceptance criteria that delivers the core value proposition. Documents what is deferred and why.

**Required fields:**

| Field | Type | Description |
|---|---|---|
| `title` | `str` | Artifact title |
| `summary` | `str` | Summary of the MVP definition |
| `mvp_scope` | `tuple[str, ...]` | Features and capabilities included in the MVP (at least one) |
| `acceptance_criteria` | `tuple[str, ...]` | Observable conditions for MVP success (at least one) |

**Optional fields (with defaults):**

| Field | Type | Default | Description |
|---|---|---|---|
| `deferred_scope` | `tuple[str, ...]` | `()` | Features explicitly deferred to post-MVP iterations |
| `milestones` | `tuple[str, ...]` | `()` | Named delivery milestones within the MVP |
| `schema_version` | `str` | `"1.0"` | Inherited from base |
| `status` | `str` | `"draft"` | Inherited from base |
| `open_questions` | `tuple[str, ...]` | `()` | Inherited from base |
| `assumptions` | `tuple[str, ...]` | `()` | Inherited from base |

**AI filling guidance:** Read `08-validation.json` `verdict` first — if `"rejected"`, MVP scoping must not proceed until blockers are resolved. Read `06-features.json` `must_have`; every must-have feature should appear in `mvp_scope` unless explicitly deferred with a justification in `deferred_scope`. Each `acceptance_criteria` entry should be directly testable. `deferred_scope` is a commitment, not a wish list — it should name exactly what is excluded and why.

## Handoff Rules

The following rules govern handoffs between methodology roles. They are derived from `docs/VOYAGE_METHOD_LOCK.md` §7.

### H01 — Interviewer → Business Analyst

| Item | Value |
|---|---|
| Artifact handed off | `01-discovery.json` |
| Readiness criteria | All required questions answered; discovery artifact validates against schema |
| Stop condition | Stakeholder undefined or scope unbounded |

### H02 — Business Analyst → UX Architect

| Item | Value |
|---|---|

### General handoff principles

- Each role consumes all earlier artifacts before producing its own.
- Downstream roles must not silently overwrite upstream decisions.
- Assumptions and open questions from earlier artifacts must be acknowledged, not discarded.
- Any contradiction identified by a downstream role must be documented in `open_questions` or escalated to the source role before handoff proceeds.
- The Consistency Validator (Step 8) is the only role whose sole purpose is cross-artifact checking — earlier roles focus on their own output and may leave gaps for the validator to catch.

## Event Lifecycle

VOYAGE-ARTIFACT-SCHEMA-1 added three `EventType` values to record methodology lifecycle events in the `EventEngine`:

| EventType name | Value | Semantic intent |
|---|---|---|
| `ARTIFACT_PRODUCED` | `"artifact_produced"` | A methodology artifact (00–09) has been produced and accepted by its owning role |
| `HANDOFF_ISSUED` | `"handoff_issued"` | A role-to-role handoff has been formally issued (artifact passed to the next role) |
| `PHASE_CLOSED` | `"phase_closed"` | A methodology phase has been formally closed (verdict issued by Auditor) |

These `EventType` values are defined in `voyage_framework/core/models.py`. This README documents their semantic intent only. Event emission is not implemented in v4.3.0 — the values are reserved for future methodology event logging.

## Out of Scope

The following are explicitly out of scope for this documentation and for any work driven by it:

- **JSON Schema export** — `model_json_schema()` generation or standalone `.schema.json` files
- **CLI validation command** — `voyage artifacts validate` or similar
- **ContextBuilder integration** — methodology artifacts are not consumed by `ContextBuilder`
- **`voyage init-project`** — project workspace bootstrap
- **Dashboard** — read-only visualization of artifact state
- **Per-artifact markdown files** — individual `docs/artifacts/00-project-intake.md` etc.
- **Example JSON files** — concrete filled artifact examples (deferred to VOYAGE-ARTIFACT-SCHEMA-3)
- **Automatic artifact generation** — AI-driven auto-fill without human review
- **Database migrations** — artifact data is not stored in SQLite in v4.3.0
- **`.voyage` workspace bootstrap** — methodology artifacts are project-owned; bootstrap is a future phase
